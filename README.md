# SQL Risk Lab

SQL Risk Lab is a public-safe PostgreSQL project for investigation-style risk analysis. It uses synthetic platform-risk data and a public Olist marketplace companion case study to demonstrate SQL queries, review queues, explainable scoring, and interpretation guardrails.

The workflow is deliberately analyst-shaped: model entities, generate controlled data, run investigation queries, materialize review queues, and document what the signals can and cannot support.

The synthetic lab models a generic hardware-enabled platform with account, order, activation, transfer, support, abuse-report, chargeback, shared-identifier, and fake-watchlist data. The `assets` table is intentionally generic, so it can stand in for a shipped device, provisioned hardware unit, marketplace-controlled item, or other trackable product without mapping to a real company or product line.

Portfolio page: https://wksprojects.com/sql-risk-lab/

## Project Snapshot

- Synthetic PostgreSQL lab for explicit review scenarios: fraud, abuse, diversion, chargeback, shared identifiers, and fake watchlists.
- Real-data Olist companion case study for marketplace integrity and operational triage.
- 15 reusable synthetic-lab SQL queries.
- 8 Olist case-study output files.
- Olist outputs cover `99,441` orders, `3,095` sellers, `8,349` medium-plus queued orders, `14` high-priority sellers, and `266` medium-priority sellers.
- Review queues use transparent flags and priority scores, not black-box conclusions.

## Scope

This lab focuses on SQL patterns used in investigations and trust and safety work:

- Shared identifiers across accounts: payment, phone, email, device, shipping address, support language, and reseller identifiers.
- Hardware assets shipped to one synthetic country and activated in another.
- Bulk orders within short time windows.
- Assets transferred shortly after purchase.
- Repeated abuse reports tied to accounts, assets, regions, and time windows.
- Chargebacks grouped by account age, country, order channel, and shipping node.
- Latest activation and latest observed location per asset.
- Diversion-style patterns: legal purchase path followed by suspicious transfer, activation, resale, or movement into high-risk synthetic geographies.
- Fake watchlists for restricted countries, reseller entities, identifiers, and email domains.

## Public-Safe Positioning

All records are synthetic. Country names, entities, account identifiers, payment identifiers, device identifiers, addresses, resellers, and watchlist entries are invented for demonstration. The project does not include real customer data, real sanctions lists, real denied-party data, real geolocation intelligence, or instructions for evading controls.

The watchlist tables are inspired by compliance workflows but use fake countries and fake entities only.

## Data Used

The project uses two data tracks:

- Synthetic lab data generated by [`scripts/generate_synthetic_data.py`](scripts/generate_synthetic_data.py). These CSVs are public-safe and committed under [`data/generated/`](data/generated/).

## SQL Techniques Demonstrated

- Multi-table joins across account, order, asset, activation, transfer, support, abuse, chargeback, reseller, country, and watchlist context.
- Common table expressions for staged investigations and scoring logic.
- Window functions for latest activation and first transfer analysis.
- Rolling 24-hour order windows for bulk-order review.
- Identifier clustering across payment, device, phone, email, address, support language, and reseller values.
- Deterministic risk scoring using transparent flags and priority bands.
- Public real-data rollups for seller quality, route risk, category risk, low-review patterns, and payment/installment review.

## Database Recommendation

PostgreSQL is the recommended database for this project. It is a better fit than SQLite because the lab uses:

- Window functions for latest activation and rolling order windows.
- Common table expressions for multi-step investigation logic.
- Date intervals for rapid-transfer and bulk-order checks.
- Views and materialized review outputs.
- Array and string aggregation for explainable risk flags.

SQLite would be more portable for a lightweight demo, but PostgreSQL better reflects real analyst workflows and leaves room for a later Streamlit dashboard.

## Core Tables

- `accounts`: synthetic account profiles and reusable identifiers.
- `reseller_profiles`: fake reseller metadata and status.
- `countries`: fake countries with synthetic risk tiers and restricted flags.
- `shipping_nodes`: fake fulfillment nodes.
- `orders`: order header records with channel, shipping, payment, and device context.
- `order_items`: asset-level order lines.
- `assets`: synthetic asset inventory and current holder.
- `activations`: asset activation events by account, country, IP, and device.
- `asset_transfers`: account-to-account asset movement events.
- `support_tickets`: support interactions with reason, language, country, and message fingerprint.
- `abuse_reports`: misuse and abuse reports tied to accounts, assets, countries, and time windows.
- `chargebacks`: payment disputes tied to orders and accounts.
- `watchlist_entities`: fake restricted countries, reseller entities, email domains, and identifier fingerprints.

## SQL Query Library

The query library moves from basic profiling to investigation-ready patterns:

1. Dataset profile and coverage checks.
2. Latest activation per asset.
3. Shared identifier clusters.
4. Account network joins across payment, device, and shipping address.
5. Ship-to versus activate-in mismatch.
6. Bulk ordering within 24-hour windows.
7. Rapid asset transfers after purchase.
8. Abuse report hotspots.
9. Chargebacks by account age, country, order channel, and shipping node.
10. Watchlist hits using fake entities only.
11. Diversion-style asset movement paths.
12. Review queue triage.
13. Asset lifecycle timeline.
14. Support language and message fingerprint clustering.
15. Risk score explainability.

## Risk Scoring

Risk scoring is intentionally transparent. SQL views create account-level and asset-level flags, then the review queue sums weighted indicators into a priority score. Each queued item includes:

- `entity_type`
- `entity_id`
- `priority_score`
- `priority_band`
- `risk_flags`
- supporting counts and timestamps

The score is not a model and should not be treated as ground truth. It is a deterministic triage aid for surfacing records that deserve human review.
